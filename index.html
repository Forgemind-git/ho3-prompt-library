What this is

A prompt library is not a folder of prompts. It is a folder of prompts that you can prove are better — because next to each one sits the lazy version you used to type, and the mediocre thing it gave you back. Keep the bad output. Without it you have a pile of text; with it you have evidence. Every prompt is written with CRISP — five parts, each doing a job: Context (the situation, and who it's for), Role (who Claude is being — an expert, not a generalist), Instruction (the task, in steps), Spec (format, length, tone, the bans) and Performance (the bar — how you'll know it's good enough to send).

Pick a problem statement

Pick the one closest to your job — the prompts only pay you back if they are for work you genuinely repeat. Each sample is a finished library of 10 prompts, and every one shows the lazy prompt beside the CRISP one so you can see exactly what the structure buys. Feel free to build a library for any other role of your own. To see the finished shape, open its sample using the link on each card.

How it works

List 8–12 things you write again and again. Not clever things — repetitive things. If you only write it twice a year, it does not belong in the library.

Write the lazy prompt first, exactly as you'd normally type it. One line, no thought. Run it, and keep the output. That mediocre answer is the most valuable thing on the page — it is what proves the rest.

Rewrite it with CRISP — Context, Role, Instruction, Spec, Performance. Leave the placeholders in square brackets so it is reusable, not a one-off.

Run the CRISP version on the same task. Same model, same day. The only thing that changed is what you told it.

Save both outputs together, with one line on why the first one was weak. Repeat until you have 8–12.

Push the library to ho3-prompt-library — a README index, one file per prompt, before/after in each.

The mistake everyone makes here
  • Skipping the lazy prompt because you already know the CRISP one is better. Then the library is just assertions — you have shown nobody anything, least of all yourself.
  • Writing prompts you'll never use. Twelve beautiful prompts for work you don't do is twelve pieces of homework. Eight for work you do weekly is a tool you'll still be using next year.
  • Leaving out Performance. It's the one people drop, and it's the one that decides whether the output is sendable — not merely correct.
The three prompts

You are building prompts, so naturally you should use prompts to do it. These three cover the whole hands-on — and the second one is the one people forget, because it feels like extra work right up until someone asks you "is this actually better?"

1 · Turn a task into a CRISP prompt
The workhorse. Run it once per prompt in your library.
Help me write a reusable CRISP prompt for a task I do again and again.

The task: <what you write, e.g. "replying to an angry customer">
My role:  <e.g. "customer support team lead at a B2B SaaS company">

Build it with all five CRISP elements, and make each one earn its place:
- Context     — the situation and who it's for
- Role        — who Claude should be (an expert, not a generalist)
- Instruction — the task, as numbered steps
- Spec        — format, length, tone, and what it must NOT do
- Performance — the bar: how I'll know the output is good enough to send

Leave every variable as a [SQUARE BRACKET] placeholder so I can reuse it.
Then show me the finished prompt as one block I can copy.
The placeholders are what make it a library rather than a one-off. A prompt with your customer's real name baked into it can only ever be used once.
2 · Prove it — the before/after
Run this in a fresh chat, or the first answer will be contaminated by everything you just said.
I'm going to give you the same task twice, and I want to see the difference.

FIRST, answer this exactly as written, with nothing added:
<paste the lazy one-line prompt>

Then STOP. Show me that answer.

Now answer the same task again, using this instead:
<paste your CRISP prompt, placeholders filled in>

Finally, in 2–3 lines: what specifically is missing from the first answer that
the second one has? Be concrete — name the things, don't just say it's "better".
Do this in a fresh chat. If you ask for the lazy answer in the chat where you just designed the CRISP prompt, Claude already knows what you want — it will quietly write a good answer to a bad prompt, and your before/after proves nothing.
3 · Write the library index
Once all 8–12 are done. A library nobody can navigate is a folder.
Here are the prompts in my library: <paste the titles and one-line purposes>

Write me a README.md index that a colleague could use on their first day:
- One line on what this library is and who it's for
- A table: prompt name | what it's for | when to reach for it
- A short "how to use these" note — fill the [BRACKETS], run it, keep the output

Keep it plain. No marketing language.
Write this last, when you know what's actually in the library — not first, from what you intended to build.
Ship it to GitHub

One file per prompt, each one carrying its own evidence. A reviewer should be able to open any single file and see the whole story without opening anything else.

ho3-prompt-library/
├─ README.md              ← the index: what each prompt is for, and when to reach for it
├─ prompts/
│  ├─ 01-reply-to-angry-customer.md
│  ├─ 02-escalation-note.md       ← each file: purpose · CRISP breakdown · the prompt
│  └─ …                              · the lazy prompt + its weak output + why
└─ .gitignore                        · the CRISP output + the bar it had to clear

One markdown file per prompt. Inside each: purpose, the CRISP breakdown, the full prompt, then before (lazy prompt · weak output · why it's weak) and after (the real output · the bar it cleared).
